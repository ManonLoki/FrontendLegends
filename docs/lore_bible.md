# FrontendLegends 世界与玩法设定集

本文档只描述当前游戏已经实现并由数据支持的设定。人物台词、数值和规则发生变化时，应同时更新对应 JSON、运行时代码、测试与本文档。

## 一、世界基调

FrontendLegends 的世界是一座由前端技术、互联网行业和市井生活共同构成的现代江湖。开发技能被称作功法，技术社区被称作门派，项目、修复和跑腿成为任务，Token 是通行货币。叙事以技术梗和生活化对白为主，但所有梗都服务于可操作的角色成长、探索、战斗和交易。

世界地图由多个城镇、郊外、门派驻地、住宅和功能场所组成。地图位置、传送关系、人物摆放与碰撞以 `assets/Map/maps/LoreWorld/` 下的 TMX 文件为准。

## 二、角色创建与身份

玩家创建角色时确定姓名、性别和四项基础属性。四项属性合计固定为 100，每项初始为 25，可在允许范围内重新分配：

| 内部字段 | 游戏名称 | 主要作用 |
| --- | --- | --- |
| `strength` | 编码 | 提高基础攻击、负重上限，并参与招架概率 |
| `agility` | 思维 | 影响命中、闪避与战斗逃跑 |
| `constitution` | 架构 | 提高防御、体力上限、冥想效率和暴击率 |
| `wisdom` | 灵感 | 提高学习与练功效率，并作为部分拜师条件 |

角色从 18 岁开始。新角色拥有 50 Token，潜能、经验和精力修为为 0，不会任何技能，也不会自动装备基础功法。食物与饮水容量为 `200 + 编码×10`。

外貌值由架构计算；当编码达到 40 且为四项最高属性时，外貌会额外降低 10 点。这一规则只影响角色资料，不改变战斗公式。

## 三、五类功法主题

技能按五个主题组织：

| 主题 | 基础功法 | 高级功法方向 |
| --- | --- | --- |
| 编码 | 基础编码 | 攻击与伤害招式 |
| 思维 | 基础思维 | 身法、闪避与行动能力 |
| 架构 | 基础架构 | 内功、精力修为与门派绝招 |
| 招架 | 基础招架 | 格挡与减伤招式 |
| 知识 | 读书写字 | 通用知识能力 |

每个主题拥有独立的基础功法槽位和门派功法槽位。学会功法不等于装备功法；玩家必须在功法界面主动装备，战斗加成、冥想条件和招式解锁才会生效。

## 四、四大门派

### NG 神教

以模板语法、组件生成、依赖注入和路由守卫为四门主要功法。门派强调结构、约束和完整工程体系。入门师父是大漠穷秋，更高层师父依据教学表提供不同等级上限。

### 香草派

崇尚原生 Web 技法，以 DOM 直拳、事件循环步、闭包内功和阻默封手为主要功法。九日、Douglas Crockford、John Resig、Brendan Eich 等人物属于该门派。

### 量子仙宗

以阶式真言、虚影掌、状态归一和边界护法为主要功法，强调声明式界面、状态与边界。门中人物包括 Pete Hunt、Tanner Linsley、Jordan Walke 等。

### 鱿鱼山庄

以模板拆招、路由步、响应内功和常驻金身为主要功法，强调响应式与渐进式体系。维特是入门师父之一，其他师父的教学能力由技能教学表明确决定。

玩家同一时间只属于一个门派并认一位师父。转投或升级师承必须满足人物数据中的四维和技能要求；师父造诣只采用教学表的明确上限，不能由拜师门槛反推。

## 五、学习、练功与冥想

### 向师父学习

学习按固定间隔推进，每次消耗潜能；完成一级所需进度与技能等级、技能数据和灵感修正有关。灵感以 25 为中性点：高灵感降低所需成本，低灵感提高成本，修正范围被限制在 0.65～1.25。教学等级不能超过师父在教学表中的明确上限。

### 自主练功

自主练功只适用于已经学会的非架构门派功法。功法等级上限同时受对应基础功法等级和精力修为约束。练功消耗当前体力与精力，并以本项目优化后的全局时钟规则推进时间；灵感越高，每次获得的练功进度越多。

### 冥想

冥想要求同时装备已学会的基础架构与本门架构功法。每次冥想先恢复当前精力；当前精力达到上限后，如果修为未达理论上限，则清空当前精力并增加 1 点精力修为。

理论修为上限为：

```text
内功值 = 基础架构等级 + 本门架构功法等级×2
修为上限 = floor(内功值×25×架构修正)
架构修正 = clamp(1 + (floor(架构)-25)×0.02, 0.60, 1.60)
```

每点精力修为提供 1 点精力上限。冥想、学习和练功的时间推进以当前 Godot 项目实现为准，不套用参照项目的旧全局时钟行为。

## 六、角色资源与全局时间

| 资源 | 含义 |
| --- | --- |
| 体力 | 战斗生命与行动资源，归零会导致失败 |
| 精力 | 使用加力、绝招、练功、疗伤等行为的资源 |
| 伤势 | 永久压低当前有效体力上限，需通过时间或疗伤恢复 |
| 食物 | 生存资源，与饮水成对消耗 |
| 饮水 | 生存资源，与食物成对消耗 |
| Token | 交易货币 |
| 潜能 | 学习功法时消耗的成长资源 |
| 经验 | 任务与战斗成长记录 |
| 精力修为 | 决定精力上限，并间接提高体力上限 |

全局时钟以秒累计。每跨过 15 秒边界，最多消耗一对食物和饮水，并恢复 3 点体力、减轻 2 点伤势；若任一资源不足，则按可组成的资源对数量结算。每跨过 28800 秒边界，年龄增加 1。一次跨过多个边界时会完整结算，不漏掉中间刻度。

## 七、战斗规则

### 基础数值

```text
基础攻击 = floor(编码 × 编码软修正)，最少为 1
基础防御 = floor(架构 × 2 × 架构软修正)，最少为 0
基础体力上限 = floor(140 × (1 + 架构×0.025))
最终体力上限 = 基础体力上限 + floor(精力上限×0.2)
有效体力上限 = 最终体力上限 - 伤势，最少为 1
```

属性软修正均以 25 为中性点并设置上下界，避免极端属性无限放大。

### 攻击顺序

一次普通攻击按固定顺序结算：命中判定、基础招架判定、暴击判定、伤害浮动、功法招式、加力、装备效果、状态与伤势。命中失败后不会继续滚动后续随机判定。

基础命中率为 `0.72 + (攻击方思维-防守方思维)×0.02`，限制在 28%～95%。基础暴击率为 `3% + 架构×0.35%`，限制在 2%～25%。防御减伤率为 `防御/(防御+80)`。

已解锁招式的触发概率为 `25% + 招式数量×4%`，最高 55%。招架招式提供 15% 基础减伤，每超过解锁等级一级再增加 1%，最高 35%。虚弱状态使受到的伤害变为 1.3 倍。

### 加力、绝招与状态

玩家可以设定加力档位。只有当前精力足以支付完整档位时才会生效，不允许部分支付；额外伤害围绕档位数值随机波动。NPC 按自身内功值使用相同含义的加力机制。

门派架构功法达到 30 级和 80 级时分别解锁一档、二档绝招。绝招分为连击、异常、削减上限和巨额伤害四种类型，各自拥有不同的精力消耗、命中修正与倍率。战斗状态包括麻痹、虚弱和中毒，由回合系统统一推进。

### 胜负与伤势

致命战斗允许体力降至 0；非致命战斗会把最低体力保留为 1。战后伤势依据战斗中实际损失的体力结算，不因战斗过程中的体力上限变化而重复计算。装备中的致伤效果可以临时削减战斗体力上限。

## 八、物品、装备与交易

物品分为食物饮品、药物、武器、防具、鞋和饰品等用途。消耗品可以恢复生存资源、体力或处理状态，并受各自冷却时间约束。装备提供攻击、防御、命中、闪避、暴击、招架、负重或特殊战斗效果。

物品数据中的 `stackLimit` 当前保留但运行时不强制限制堆叠，这是与参照项目有意保留的差异。商店买价使用物品标价，卖价按当前交易公式折算。已经装备的物品不能直接作为任务物品交付，必须先卸下。

装备只在当前游戏会话内存在，不写入存档；重新读取存档后角色为空手状态。这是当前明确设计，不是存档缺陷。

## 九、任务体系

当前任务分为四类：

| 任务 | 发布者 | 核心玩法 |
| --- | --- | --- |
| DARK 学项目练手 | 道士 | 在电脑完成随机项目，消耗相应体力并获得固定奖励 |
| 九日环任务 | 九日 | 前往指定地图通知人物，或购买指定物品并交付 |
| 小不二悬赏 | 小不二 | 前往随机地图击败动态生成的悬赏目标 |
| 活阎王生死簿 | 活阎王 | 击败指定目标并按目标强度获得潜能 |

环任务每环 10 次。九日环的奖励随环数成长并带随机浮动；悬赏目标的属性和奖励会随轮次增长；任务冷却使用全局游戏时间。玩家当前只允许一个活动任务槽位，任务放弃功能按现设计保留，不照搬参照项目限制。

## 十、人物与对白原则

人物的显示名、年龄、门派、描述、默认对白、属性、技能、装备、货币和角色职责以 `assets/Data/npcs.json` 为准。任务对白以 `assets/Data/quests.json` 为准，商店与物品文案以 `assets/Data/items.json` 为准，功法名称和招式以 `assets/Data/skills.json` 为准。

对白保持人物原有语气，包括口语、行业黑话和带情绪的表达。代码不得另建一套同义对白覆盖数据；只有运行状态组合、数值插值和通用错误信息可以由运行时代码生成。

## 十一、稳定标识与显示文本

人物、物品、技能和地图中的拼音或既有英文 ID 是跨 JSON、TMX、存档和代码引用的稳定数据主键，不属于需要清理的“拼音变量”。代码变量、函数、类和存档键必须使用规范英文；面向玩家的内容使用中文。若未来更改数据主键，必须同时迁移地图对象、数据引用、存档和测试。
