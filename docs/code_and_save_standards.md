# 代码与存档规范

本文档规定 FrontendLegends 的源码注释、标识符和存档结构。运行时行为以 Godot 场景与 GDScript 为准，本文档必须随实现同步更新。

## 中文注释

- 项目自有源码、测试、配置说明和设计文档使用简体中文。
- 每个类、领域常量、状态字段、函数、条件分支、公式步骤和持久化写入都应说明用途或约束。
- 注释解释“为什么这样做”和“该步骤在规则中的含义”，不重复翻译语法。
- `JSON`、`Godot`、`TMX`、`HP`、`MP` 等公认技术名词可以保留英文；首次出现时应给出中文含义。
- 引用真实字段名或资源 ID 时保留原样并使用反引号标记，避免文档与运行时脱节。

## 英文标识符

- 变量、常量、函数、类名和信号名必须使用含义明确的英文，不使用汉语拼音。
- GDScript 变量、函数和存档键使用 `snake_case`，常量使用 `UPPER_SNAKE_CASE`，类名使用 `PascalCase`。
- 人物、物品、技能和地图的资源 ID 是数据主键，不属于“拼音变量”。已有主键若需改名，必须提供全量引用迁移，不能只改显示文本。
- 外部数据中为对齐参照项目保留的 `camelCase` 字段属于输入协议；运行时代码只在数据边界读取，不把该风格扩散到新存档字段。

## 文件职责

- 项目源码以 300 行为拆分目标，500 行为绝对上限。
- 单个模块只负责一个领域，例如技能学习、技能装备、技能训练或技能评级。
- 主协调器只组合窄接口，不重复实现领域公式。
- 新增逐行注释前先拆分接近上限的文件，不能通过压缩语句规避行数限制。

## 存档版本

当前存档版本为 `3`，文件路径仍为 `user://frontend_legends_save_v2.json`，以便原有玩家无感升级。文件名中的版本仅是历史名称，不再代表结构版本。

顶层字段如下：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `version` | 整数 | 存档结构版本 |
| `profile` | 字典 | 角色身份、基础属性、生命资源和技能状态 |
| `game_time_sec` | 浮点数 | 游戏全局时钟累计秒数 |
| `combat_state` | 字典 | 当前体力、精力与伤势 |
| `inventory` | 字典 | 物品 ID 到持有数量的映射 |
| `item_cooldowns` | 字典 | 物品 ID 到剩余冷却时间的映射 |

技能状态字段如下：

| 字段 | 含义 |
| --- | --- |
| `levels` | 技能 ID 到等级的映射 |
| `equipped_basic` | 主题到已装备基础功法 ID 的映射 |
| `equipped_special` | 主题到已装备门派功法 ID 的映射 |
| `progress` | 保留的通用进度映射 |
| `learn_progress` | 向 NPC 学习时的累计进度 |
| `practice_progress` | 自主练功时的累计进度 |
| `force_power` | 玩家当前设定的内力输出档位 |

## 旧档迁移

- 版本 `2` 和版本 `3` 均可读取。
- `learnProgress` 自动迁移为 `learn_progress`。
- `practiceProgress` 自动迁移为 `practice_progress`。
- `forcePower` 自动迁移为 `force_power`。
- 新旧字段同时存在时采用新字段，并删除旧字段，避免双数据源。
- 保存时始终写出版本 `3` 和标准字段。

## 验证要求

- 每次更改存档结构必须补充旧档读取和新档回写测试。
- 每次拆分领域模块后必须运行无界面全量对齐测试。
- 完成中文化前必须扫描英文自然语言注释、拼音标识符和超过 500 行的源码。
