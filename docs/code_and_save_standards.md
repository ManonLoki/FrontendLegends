# 代码与存档规范

本文档规定 FrontendLegends 的源码注释、标识符和存档结构。运行时行为以 Godot 场景与 GDScript 为准，本文档必须随实现同步更新。

## 基础玩法变更门禁

- `docs/design_invariants.md` 是学习与战斗基础逻辑的规范副本；`AGENTS.md` 与 `CLAUDE.md` 对自动化修改实施同一保护。
- 普通维护只可调整数值、系数、阈值和计算公式，并必须保留学习资源身份与阶段顺序，以及战斗的四属性、已装备功法、装备和显式修正输入。
- 若变更会删除输入、交换职责、合并资源、绕过阶段、改变 `combatRank`/`combatRole` 职责或加入隐藏倍率，首次需求不视为授权。必须停止写入，列出准确文件、行为、存档/数据和测试影响，等用户在后续消息中明确一次性解锁。
- 每次允许的调参仍要同步设定文档、平衡工作簿与结构回归；测试通过不能替代用户授权。

## 中文注释

- 项目自有源码、测试、配置说明和设计文档使用简体中文。
- 每个类、领域常量、状态字段、函数、条件分支、公式步骤和持久化写入都应说明用途或约束。
- 注释解释“为什么这样做”和“该步骤在规则中的含义”，不重复翻译语法。
- `JSON`、`Godot`、`TMX`、`HP`、`MP` 等公认技术名词可以保留英文；首次出现时应给出中文含义。
- 引用真实字段名或资源 ID 时保留原样并使用反引号标记，避免文档与运行时脱节。

## 英文标识符

- 变量、常量、函数、类名和信号名必须使用含义明确的英文，不使用汉语拼音。
- GDScript 变量、函数和存档键使用 `snake_case`，常量使用 `UPPER_SNAKE_CASE`，类名使用 `PascalCase`。
- 人物、物品、技能、任务、世界事件和地图统一使用 RFC 4122 UUID 作为稳定主键。显示名、文件名和 ID 互不推导，禁止再用拼音、名称缩写或人工递增数字生成主键。
- 六份运行时数据文档的 `version` 必须一致；新增或修改主键后必须执行 `npm run data:ids:check`，校验全局唯一性、跨表引用与 TMX 声明。
- 外部数据中为对齐参照项目保留的 `camelCase` 字段属于输入协议；运行时代码只在数据边界读取，不把该风格扩散到新存档字段。

## 文件职责

- 项目源码以 300 行为拆分目标，500 行为绝对上限。
- 单个模块只负责一个领域，例如技能学习、技能装备、技能训练或技能评级。
- 主协调器只组合窄接口，不重复实现领域公式。
- 新增逐行注释前先拆分接近上限的文件，不能通过压缩语句规避行数限制。

## 存档版本

当前存档版本为 `5`，文件路径为 `user://frontend_legends_save_v5.json`。

顶层字段如下：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `version` | 整数 | 存档结构版本 |
| `profile` | 字典 | 角色身份、基础属性、生命资源和技能状态 |
| `game_time_sec` | 浮点数 | 游戏全局时钟累计秒数 |
| `combat_state` | 字典 | 当前体力、精力与伤势 |
| `inventory` | 字典 | 物品 ID 到持有数量的映射 |
| `item_cooldowns` | 字典 | 物品 ID 到剩余冷却时间的映射 |

技能状态字段如下：

| 字段 | 含义 |
| --- | --- |
| `levels` | 技能 ID 到等级的映射 |
| `equipped_basic` | 主题到已装备基础功法 ID 的映射 |
| `equipped_special` | 主题到已装备门派功法 ID 的映射 |
| `progress` | 保留的通用进度映射 |
| `learn_progress` | 向 NPC 学习时累计的固定口径学习经验 |
| `learn_potential_spent` | 当前等级已经实际消耗的潜能，用于结算 Token 学费 |
| `practice_progress` | 自主练功时的累计进度 |
| `force_power` | 玩家当前设定的内力输出档位 |

## 版本边界

- v5 将全部持久化资源引用切换为 UUID，因此 v2、v3、v4 存档全部作废。
- 读取器只接受 `version == 5`；不存在字段别名、数值换算、旧目录探测或旧文件复制逻辑。
- 保存时始终写出版本 `5` 和标准蛇形字段。版本不匹配时玩家必须重新创建角色。

## 初始化与派生值

- 自动加载依赖顺序固定为 `DataRegistry → SkillSystem → GameState`，之后再加载背包、任务、NPC、战斗和结算系统。
- 读档、卸下功法、战败降级等会改变体力或精力上限的路径，必须重算派生四维并调用战斗状态规范化，把当前值钳制到新上限。
- 测试必须使用 `GameState.use_test_save_path(...)` 写入系统临时目录，不得读取或覆盖正式 `user://` 存档。

## 验证要求

- 学习公式变化必须通过 `tests/alignment/learning_suite.gd`，证明固定经验逐级增长、灵感只影响潜能转换和实际 Token 成本。
- 战斗公式变化必须通过 `tests/combat/invariant_contract.gd` 及两套战斗入口测试，证明四属性、已装备功法、装备和显式修正仍生效且阶位/原型没有越权。
- 每次更改存档结构必须补充当前版本回写测试和旧版本拒绝测试。
- 每次更改自动加载顺序或生命资源公式必须执行隔离 `user://` 的冷启动测试。
- 每次拆分领域模块后必须运行无界面全量对齐测试。
- 完成中文化前必须扫描英文自然语言注释、拼音标识符和超过 500 行的源码。
